# Future Feature — Per-user (anonymous) aggregation for /usage-report

> Status: **design only — not yet implemented**
> Related issue: [#27](https://github.com/lakhi/health-research-agent-api/issues/27) — NeX: Implement Anonymous Usage Based Metrics

## Context

The `/usage-report` skill currently answers "how many **sessions** used the agent" but not "how many **distinct people** came back, and how long did each of them spend". The existing `agent_usage_metrics` schema stores only `anonymous_session_id`, a per-visit UUID generated by the frontend — there is no longer-lived identifier that links multiple sessions together. The team wants reports that show, per recurring visitor:

- Total number of sessions
- Total number of requests
- Total tokens + total EUR cost
- **Total time spent** (sum of session elapsed times across all their sessions)
- First-seen / last-seen dates

Privacy constraint: the system is deliberately identity-free (`db/models/usage_metrics.py:22-23`). The solution must stay anonymous — we introduce a **second** random UUID with longer lifetime (browser localStorage), not any real user identity.

Frontend lives in a separate repo: [`github.com/lakhi/health-research-agent-ui`](https://github.com/lakhi/health-research-agent-ui). This plan covers the **backend + DB + report changes only**, and documents the exact contract the frontend must implement later.

## Design decisions (locked in)

| Decision | Choice |
|---|---|
| DB column name | `anonymous_user_id VARCHAR(64) NULL` (mirrors `anonymous_session_id`) |
| Request field name | `user_id` (mirrors the existing asymmetry where request field is `session_id` but column is `anonymous_session_id`) |
| Single-request session time | Fall back to `duration_seconds` (agent compute time) so one-shot visitors aren't invisible in the "time spent" column |
| Schema migration | Nullable column + new idempotent `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` SQL file. No Alembic. Historical rows stay NULL and are excluded from per-user queries via `WHERE anonymous_user_id IS NOT NULL` |
| Backwards compatibility | All existing queries and reports unchanged. New queries are additive |

### 3. Reports — two new queries

Append to `scripts/sql/usage_reports.sql`:

**Q9 — Unique users summary**
```sql
-- 9. Unique users and engagement ratios
SELECT
    COUNT(DISTINCT anonymous_user_id) AS unique_users,
    COUNT(DISTINCT anonymous_session_id) AS total_sessions,
    ROUND(
        COUNT(DISTINCT anonymous_session_id)::numeric
        / NULLIF(COUNT(DISTINCT anonymous_user_id), 0),
        2
    ) AS avg_sessions_per_user,
    ROUND(
        COUNT(*)::numeric
        / NULLIF(COUNT(DISTINCT anonymous_user_id), 0),
        2
    ) AS avg_requests_per_user
FROM agent_usage_metrics
WHERE date BETWEEN :from_date AND :to_date
  AND anonymous_user_id IS NOT NULL;
```

**Q10 — Per-user rollup with total time spent** (uses a CTE for the single-request fallback):
```sql
-- 10. Per-user aggregate: sessions, requests, tokens, cost, time spent
WITH session_stats AS (
    SELECT
        anonymous_user_id,
        anonymous_session_id,
        COUNT(*) AS messages,
        SUM(total_tokens) AS session_tokens,
        SUM(cost_eur) AS session_cost,
        CASE
            WHEN COUNT(*) = 1 THEN COALESCE(MAX(duration_seconds), 0)
            ELSE EXTRACT(EPOCH FROM MAX(created_at) - MIN(created_at))
        END AS session_time_seconds,
        MIN(date) AS session_date
    FROM agent_usage_metrics
    WHERE date BETWEEN :from_date AND :to_date
      AND anonymous_user_id IS NOT NULL
    GROUP BY anonymous_user_id, anonymous_session_id
)
SELECT
    anonymous_user_id,
    COUNT(DISTINCT anonymous_session_id) AS sessions,
    SUM(messages) AS requests,
    SUM(session_tokens) AS total_tokens,
    ROUND(SUM(session_cost)::numeric, 4) AS total_cost_eur,
    ROUND(SUM(session_time_seconds)::numeric, 0) AS total_time_spent_s,
    MIN(session_date) AS first_seen,
    MAX(session_date) AS last_seen
FROM session_stats
GROUP BY anonymous_user_id
ORDER BY requests DESC;
```

### 4. Frontend contract (for the UI repo, documented only here)

The `health-research-agent-ui` repo will need a separate PR implementing this contract. Document in the backend PR description:

| Field | Value |
|---|---|
| Request body field | `user_id` (also accepted via multipart form-data with the same name) |
| Format | UUID v4 lowercase string, 36 chars |
| Lifetime | Generate once on first visit, persist indefinitely in `localStorage` |
| Suggested localStorage key | `health_research_anonymous_user_id` |
| Relationship to `session_id` | `session_id` stays per-visit/per-conversation (status quo); `user_id` is strictly longer-lived |
| Backwards behaviour | Backend treats missing `user_id` as NULL — old clients keep working; their rows simply won't appear in per-user reports |
